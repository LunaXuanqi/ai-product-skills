# UI / 体验 Review 清单

双用途工具：
1. **做 demo 前** — 喂给 Claude Code 当生成约束
2. **做完后** — 自己过一遍当验收 gate

本文档只管成品对不对。做什么方向由 `PRINCIPLES.md` 定，怎么做由 `SOP.md` 管。

---

## 怎么用

### 用法一：喂给 CC 当生成约束（做之前）

在 Claude Code 里开始做 demo 前，把本文档贴给 CC，让它按 checklist 生成，不是生成完再改。最省力的模板：

```
我要做一个 [方案简述]。请先读 REVIEW.md，严格按五层 checklist 生成。完成后逐层自检，未满足的条目说明原因。
```

### 用法二：自己过一遍当验收 gate（做之后）

demo 出来后，人工逐层过。CC 可能漏，必须两道闸。

---

## 五层 Checklist

| 层 | 对应时段 / 功能 |
|----|----------------|
| 🧠 心智层 | 用户打开 / 看到的前 3 秒 |
| 👀 感知层 | 用户眼里的视觉品质（静态看起来如何） |
| 🖱️ 交互层 | 用户每次操作时的反应 |
| 🌊 流程层 | 用户完整走一遍的顺畅度 |
| 🩹 容错层 | 出事了之后怎么办 |

---

### 🧠 心智层

- [ ] 用户 3 秒内能看懂这是什么、能干什么
- [ ] 主行动点一眼可见，不用找
- [ ] 没有让用户困惑的术语或图标

**不合格信号**：用户问"这是干嘛的？"、"我该点哪？"

---

### 👀 感知层

- [ ] 视觉层次清晰（主次分明）
- [ ] 间距、对齐、字号一致
- [ ] 颜色使用有意义（不是装饰）
- [ ] 空状态、加载态有设计
- [ ] 默认内容高大上，拒绝廉价感

**不合格信号**：看起来"糙"、"乱"、"像没做完"

---

### 🖱️ 交互层

响应时间（供参考，不强制）：
- 即时反馈：< 100ms
- 短操作：< 1s
- 长操作：需进度指示

**硬要求**（必须做到）：
- [ ] 点击后有即时反馈（状态变化、loading、禁用）
- [ ] 不能连点导致重复提交

**手感（软指标）**：
- [ ] 动效自然，不突兀
- [ ] 可点区域足够大
- [ ] hover/press 状态清晰

**不合格信号**：点了没反应、不知道在加载、动画卡顿

---

### 🌊 流程层

- [ ] 主流程最短路径（能少一步就少一步）
- [ ] 步骤之间有明确指引
- [ ] 用户知道自己在哪、下一步是什么
- [ ] 能返回、能撤销

**关于 aha 点**：
不用"X 秒以内"作为一刀切。看用户是否在心理窗口内等到 aha：
- 简单需求（改个字）：秒级
- 中等需求（生成页面）：10-30 秒可接受，需进度
- 复杂需求（生成整个应用）：分钟级可接受，需阶段反馈

**不合格信号**：用户迷路、不知道下一步、想返回找不到路

---

### 🩹 容错层

- [ ] 错误信息说人话（不是报错码）
- [ ] 告诉用户怎么解决，不只是"出错了"
- [ ] 关键操作可撤销
- [ ] 数据不会意外丢失

**不合格信号**：用户出错后不知道怎么办、数据丢了

---

## 8 条原则扫尾

最后用 `PRINCIPLES.md` 的 8 条原则扫一遍：

1. [ ] **最短路径** — 还能再少一步吗？
2. [ ] **秒级 aha** — 用户多快能看到价值？
3. [ ] **修改优先** — 改起来顺不顺？
4. [ ] **需求先于打磨** — 核心需求满足了吗？
5. [ ] **默认高大上** — 分享出去丢人吗？
6. [ ] **Demo 优先** — 这个方案能直接看懂吗？
7. [ ] **先跑再推** — 有没有过度设计？
8. [ ] **为未来设计** — 模型升级后方案还成立吗？
