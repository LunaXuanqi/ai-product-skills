# 产品方案 SOP

从"脑子里一个想法"到"团队能执行"的标准动作路径。

本文档只管过程。做什么方向由 `PRINCIPLES.md` 定，成品对不对由 `REVIEW.md` 判。

---

## 前置共识

在动笔之前，先认同三件事：

1. **Demo 优先** — 能做出来看的，不写文档论证
2. **先跑再推** — 试错成本极低，半天做出来比两周论证强
3. **为未来设计** — 方案骨架不因模型升级而崩

---

## 四步流程

### Step 1: 压想法

**动作**：用一句话写清楚——**谁，在什么场景，做什么，得到什么。**

❌ 坏例子：
> "我们要做一个让 Builder 更方便管理 App 的工具"（谁都没错，但没人知道要做啥）

✅ 好例子：
> "Builder 在 App 上线第 7 天，运营 Agent 主动生成一张'你本周值得调的三件事'的 Tweaks 面板，让 Builder 一次拖拽完成运营优化。"

---

### Step 2: Demo 与 PRD 的关系

PRD 在不同时段承担两种不同角色：

| 角色 | 时机 | 作用 |
|------|------|------|
| **脚手架** | 思路没想清时 | PRD 先行，帮助理清思路 |
| **补丁** | Demo 完成后 | 记录 demo 展现不出来的东西 |

顺序由 PM 自己判断——思路还没想清时 PRD 可以先行作为脚手架，思路清晰时直接动 demo，不强制统一节奏。

#### 做 demo 的动作：
1. 把 `REVIEW.md` 喂给 Claude Code 当生成约束
2. 快速出原型，所见即所得
3. 和团队 review，收集反馈

#### 什么时候必须补 PRD（补丁角色）：
- 状态逻辑复杂（多状态切换、并发）
- 业务规则多（权限、计费、限制）
- 决策取舍需要记录（为什么不做 A 而做 B）

#### 什么时候不补 PRD：
- Demo 已经足够清晰
- 逻辑简单，研发看 demo 就能做

**核心判断**：PRD 的存在是为了想清楚或说清楚，不是为了"有文档"。如果两者都不满足，就别写。

---

### Step 3: 瘦身

**瘦身 ≠ 砍边界**。瘦身砍的是"不必要的实体"，具体四类：

| 砍什么 | 例子 |
|--------|------|
| **冗余功能** | 用户根本不会用的按钮 |
| **冗余路径** | 能一步到位的拆成三步 |
| **冗余确认** | 不需要二次确认的弹窗 |
| **过度防御** | 极小概率场景的复杂处理 |

**什么不在瘦身范围内**——必要边界（异常输入、关键状态切换、权限组合、并发冲突）。这些反而要加强，放到第四步"边界梳理"处理。

瘦身对应元原则"非必要勿增实体"，"必要"是关键词。

**瘦身节奏**：每次和团队 review 后都过一遍这四类，不是"定稿才瘦"。

---

### Step 4: 边界梳理

瘦身在砍不必要的东西；边界梳理在补必要但容易漏的东西。两个动作一起做。

#### 被质疑"边界不全"的四类原因及解法：

| 原因 | 解法 |
|------|------|
| A. 真的没想全 | 用边界 checklist 扫一遍 |
| B. 想过了但没写出来 | PRD 补丁强制包含"状态清单 + 业务规则 + 边界触发" |
| C. 研发/测试标准更严 | 和研发/测试一起建 checklist |
| D. 迭代快来不及梳理 | 边界梳理作为 PR/提测前的强制 gate |

#### 边界 Checklist（PM 方案完成后过一遍）：

- [ ] **异常输入**：空值、超长、特殊字符、非法格式
- [ ] **状态切换**：所有状态之间的转换是否都考虑到
- [ ] **权限组合**：不同角色 × 不同状态的矩阵
- [ ] **并发冲突**：多人同时操作、重复提交
- [ ] **网络异常**：断网、超时、重试
- [ ] **数据边界**：空列表、单条、满页、超大量

过不了的条目要么补进方案，要么在 PRD 补丁里明写"本版不处理，原因是..."。
